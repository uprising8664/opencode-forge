You are setting up git worktrees from a manifest file created by Architect for parallel lane execution.

## Input

The user invoked: `/setup-plan-worktrees $ARGUMENTS`

The argument is the **manifest file path** — relative to `.forge/plans/` (e.g. `api-refactor-manifest.md`).

If no arguments provided, output:
```
Usage: /setup-plan-worktrees <manifest-file>
Example: /setup-plan-worktrees PROJ-42-manifest.md

The manifest file is generated by Architect when creating parallel multi-lane plans.
Location: .forge/plans/<prefix>-manifest.md
```
Then stop.

## Manifest Format

```markdown
# {Plan Name} - Manifest

## Config
- **Prefix**: <parent-prefix>
- **Project**: <project-name>
- **Integration Branch**: <branch-name> | none
- **Base Branch**: main

## Lanes

| Wave | Prefix | Plan File | Worktree | Base | Depends On |
|------|--------|-----------|----------|------|------------|
| 0 | PROJ-43 | PROJ-43-lane-1-models.md | PROJ-43-lane-1 | main | — |
| 0 | PROJ-44 | PROJ-44-lane-2-routes.md | PROJ-44-lane-2 | main | — |
| 1 | PROJ-42 | PROJ-42-lane-3-tests.md | PROJ-42-lane-3 | integration | 1, 2 |
```

The `Prefix` column in the Lanes table is **optional**. When present, each lane uses its own prefix for plan file and worktree naming. When absent, all lanes inherit the Config `Prefix`. Either way, the `Plan File` and `Worktree` columns are the authoritative values used for worktree creation.

**Base column values:**
- `main` — branch from main
- `integration` — branch from the integration branch
- `<number>` — branch from that lane's worktree branch (chained dependency)

## Workflow

### Step 1: Read manifest

```bash
cat .forge/plans/${MANIFEST_FILE}
```

Parse the **Config** section for `Integration Branch` and the **Lanes** table for all rows.

### Step 2: Create integration branch (if specified)

If Integration Branch is not `none`:
```bash
# Pull latest on main (run from inside the main worktree)
git pull --rebase --autostash
# Create integration branch from main — no checkout needed in a bare repo layout
git branch <integration-branch> main
git push -u origin <integration-branch>
```

### Step 3: Check existing worktrees

List current worktrees and check for conflicts:
```bash
git worktree list
```

Compare the output against the Lanes table. If any worktree paths or branch names already exist, use the `question` tool to ask the user:
> Worktrees already exist: `<list>`. How should I proceed?
> - **Skip** existing, create only missing ones
> - **Delete and recreate** all (uses `git worktree remove <path>` then recreates)

### Step 4: Create worktrees by wave

Process waves in ascending order (0, 1, 2…). Within each wave, create all worktrees as siblings of the main worktree:

```bash
# Base = main — branch from main
git worktree add ../<WORKTREE> -b <WORKTREE> main

# Base = integration — branch from the integration branch
git worktree add ../<WORKTREE> -b <WORKTREE> <integration-branch>

# Base = <number> (chained — branch from another lane's worktree branch)
git worktree add ../<WORKTREE> -b <WORKTREE> <other-lane-branch>
```

All paths use `../` because worktrees are created as siblings to the current (main) worktree in a bare repo layout.

### Step 5: Distribute plan files

For each lane row, copy its plan file into the worktree:

```bash
mkdir -p ../<WORKTREE>/.forge/plans
cp .forge/plans/<PLAN_FILE> ../<WORKTREE>/.forge/plans/
cp .forge/plans/<MANIFEST_FILE> ../<WORKTREE>/.forge/plans/
```

**Why**: `.forge/` is gitignored, so new worktrees won't have it. Each worktree needs its plan file for `/start-work` to function.
